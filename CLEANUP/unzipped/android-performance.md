# Android performance

The Android-specific performance layer. Three goals, in the order users feel them: install fast, load fast, smooth on every screen. Each has a measurement, a set of usual causes, and a fix. Do not guess — find the exact cause with a trace, change one thing, then prove the change with numbers.

Contents:
- Measure first, always
- Goal 1: load fast (cold start)
- Goal 2: install fast (app size)
- Goal 3: smooth on every screen (jank)
- Finding the exact cause

## Measure first, always

A guess is not a finding. Before changing code, capture a number you can compare against later.

- **Macrobenchmark** runs on a real device, not the emulator, and measures cold-start time, frame timing, and scroll jank. JIT and install behaviour differ on real hardware, so emulator numbers lie. Add a `:macrobenchmark` module (Android Studio: New Module → Benchmark → Macrobenchmark) and write a startup test and a scroll test for the screens that matter.
- **Perfetto / system trace** records what ran on each thread during the slow moment. This is how you see the exact method blocking the main thread, not a vague "startup is slow."
- **ApplicationExitInfo** (API 30+) tells you why the previous process died — ANR, native crash, or a low-memory kill that never reached a normal crash reporter. Read it on the next launch to catch stability problems standard reporters miss.
- **Play Console → Android Vitals** gives the production targets: ANR rate, crash rate, excessive-wakeup and slow-start thresholds. Treat a screen that exceeds a Vitals threshold as a blocking issue, because Play demotes apps that do.

Write down the before number. Write down the after number. A fix without both is a hope.

## Goal 1: load fast (cold start)

Cold start is the time from tap to the first usable frame. Aim under ~500ms on a mid-range device; anything over a second feels broken.

Usual causes in vibe-coded apps:

- **Heavy work in `Application.onCreate`.** Initialising analytics, crash reporting, image loaders, dependency graphs, and databases synchronously at launch blocks the first frame. This is the most common cold-start killer.
- **Eager content-provider initialisers.** Libraries auto-initialise through a `ContentProvider` before your code runs. Several of them stack up before the first screen.
- **Main-thread disk or network I/O at launch.** Reading a large file, a config blob, or a synchronous database query during startup.

Fixes:

- **Defer and lazy-init.** Use the AndroidX **App Startup** library to control and lazy-load component initialisers, and move non-essential setup off the launch path. Initialise a thing the first time it is actually used, not at boot.

```kotlin
// Incorrect — everything runs before the first screen can draw
class App : Application() {
  override fun onCreate() {
    super.onCreate()
    Analytics.init(this)        // network
    ImageLoader.warmUp(this)    // disk
    Database.migrateNow(this)   // disk + CPU
  }
}

// Correct — only the essential runs now; the rest is deferred or lazy
class App : Application() {
  override fun onCreate() {
    super.onCreate()
    CrashReporter.init(this)    // small, needed early
  }
}
// Analytics, image loader, and DB migrate on first use, off the main thread,
// or via App Startup with a lazy initializer.
```

- **Add a Baseline Profile.** A Baseline Profile lists the classes and methods used during startup and the first screens. Play ships it with the app, and ART ahead-of-time-compiles those methods at install, before the user opens the app. Google reports 30–40% faster cold start and smoother first-launch scrolling. The release build must have `isMinifyEnabled = true`; R8 rewrites the profile rules to match the obfuscated code. The profile is generated by a Macrobenchmark test and bundled by the Baseline Profile Gradle plugin (the binary `baseline.prof` must stay under 1.5 MB).
- **Do not fake it with a long splash.** A splash screen that hides a slow start is still a slow start. Use the AndroidX SplashScreen API correctly and fix the work behind it.

## Goal 2: install fast (app size)

Download and install time track app size. Smaller is faster to install and frees device storage.

- **Ship an Android App Bundle (AAB), not a universal APK.** Play generates per-device APKs split by screen density, ABI, and language, so each user downloads only what their device needs.
- **Turn on R8 in full mode.** Set `isMinifyEnabled = true` and `isShrinkResources = true` in the release build. R8 removes unused code, optimises, and obfuscates, and resource shrinking drops unused resources. Real apps see roughly 10–15% smaller APK/AAB, faster startup, and fewer ANRs from this alone. A release build with minification off is a blocking finding.

```kotlin
// build.gradle.kts — release
buildTypes {
  release {
    isMinifyEnabled = true
    isShrinkResources = true
    proguardFiles(
      getDefaultProguardFile("proguard-android-optimize.txt"),
      "proguard-rules.pro"
    )
  }
}
```

- **Audit the keep rules.** Overly broad `-keep` rules disable R8 on whole packages and bloat the result. Google's R8 Analyzer skill flags redundant and too-broad rules and rules that libraries already provide. Run it before a major release.
- **Cut dependency weight.** A whole library pulled in for one function is download weight. Inspect the bundle with the APK Analyzer (Android Studio: Build → Analyze APK) or Ruler to see exactly what takes space, then remove or replace the heavy items.
- **Align for the 16 KB native page size.** Recent Android requires native libraries aligned to a 16 KB page size for Play. Misaligned `.so` files block the upload — check this if the app has native code.

## Goal 3: smooth on every screen (jank)

Jank is a dropped frame the user sees as a stutter. The frame budget is about 16.6ms at 60Hz and 8.3ms at 120Hz. Miss it and the frame drops.

Usual causes:

- **Work on the main thread during a frame.** A database query, JSON parse, or bitmap decode inside a scroll or a tap handler. Any of these blows the budget.
- **Rendering a long list without recycling.** A `Column` (Compose) or a layout built by looping over a large array mounts everything at once. Use `LazyColumn` / `LazyRow`, or a `RecyclerView` in the View system, so only visible rows render.
- **Compose recomposition storms.** A composable re-runs because it reads state that changes often, or takes an unstable parameter. The whole subtree recomposes on every change.
- **Overdraw and deep layout.** The same pixel painted many times, or a view hierarchy many layers deep, costs measure and draw time every frame.

Fixes:

```kotlin
// Incorrect — non-lazy list mounts every row; reads a hot value high up
@Composable
fun Feed(items: List<Post>, scroll: ScrollState) {
  Column(Modifier.verticalScroll(scroll)) {   // all rows mount at once
    items.forEach { PostRow(it) }
  }
}

// Correct — lazy list recycles rows; stable keys and contentType help reuse
@Composable
fun Feed(items: List<Post>) {
  LazyColumn {
    items(items, key = { it.id }, contentType = { "post" }) { PostRow(it) }
  }
}
```

- **Move work off the main thread.** Do I/O on a coroutine with `Dispatchers.IO`, switch to the main dispatcher only to update UI. Never query a database or decode an image on the scroll path.
- **Stabilise Compose.** Read state as low in the tree as possible (deferred state reads), use `remember` and `derivedStateOf` for computed values, give stable types to composable parameters, and rely on Strong Skipping so unchanged composables skip recomposition. Verify with the Compose recomposition counts in Layout Inspector — do not guess which composable is hot.
- **Reduce overdraw and layout depth.** Flatten the hierarchy and remove redundant backgrounds. Toggle "Debug GPU overdraw" in developer options to see it.
- **Measure jank in production with JankStats**, and confirm fixes with the frame-timing metric in a Macrobenchmark scroll test.

## Finding the exact cause

"The app is slow" is not actionable. The procedure that turns it into a file and a line:

1. Reproduce the slow moment on a real device.
2. Record a Perfetto / system trace of exactly that moment (the cold start, or the scroll on the janky screen).
3. Open the trace and find the long span on the main thread — the frame that missed budget, or the block before the first frame.
4. Read the method name on that span. That is the cause. Trace it back to the file and line in the codebase.
5. Apply the matching fix above.
6. Re-run the Macrobenchmark test and compare the before and after numbers.

This is the investigation the agent lineup runs (see `references/agent-lineup.md`): one agent reproduces and traces, one locates the code, one fixes, one verifies the numbers.
