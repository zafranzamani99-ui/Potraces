## 1. CONCEPT — Directions, Statuses, Semantic Colors

### 1.1 The two directions (`DebtType`)
`src/types/index.ts:827` — `export type DebtType = 'i_owe' | 'they_owe';`

- **`i_owe`** — money I owe to someone else (my responsibility / liability).
- **`they_owe`** — money someone else owes me (an asset / receivable).

## 3. PAYMENTS

### 3.1 `Payment` model
See §1.3. Key: `walletId` (which wallet was used), `linkedTransactionId` (the income/expense Transaction created for this payment), `tipAmount` (overpayment beyond remaining), `editLog`.

### 3.2 Store: `addPayment`
`debtStore.ts:126–163`:
- Rejects `amount <= 0` (returns `null`) and rejects if debt is already `settled` (returns `null`).
- Computes `remaining = max(0, totalAmount - paidAmount)`; returns `null` if `<= 0`.
- **Caps** the payment: `cappedAmount = min(amount, remaining)` — a debt can never be overpaid in the store; overpayment is surfaced as a **tip** at the UI layer (see §3.4).
- Appends payment, recomputes `paidAmount = min(totalAmount, paidAmount + cappedAmount)`, re-derives status (`settled` if `>= total`, else `partial` if `> 0`, else `pending`). Returns the payment id.

### 3.3 Partial vs full
Status is purely derived from `paidAmount` vs `totalAmount` (`debtStore.ts:148–151`, also recomputed in `updateDebt`, `deletePayment`, `updatePayment`). Partial payment ⇒ `partial`; reaching/exceeding total ⇒ `settled`.

### 3.4 Wallet + linked-transaction integration (the core flow)
`processPayment` in `DebtTracking.tsx:1518–1638`. For a single debt:
1. Re-reads the fresh debt from the store; aborts if settled/archived; validates the chosen wallet still exists (`:1520–1536`).
2. Computes tip = overpayment beyond `remainingAmount` (`:1538–1539`).
3. Calls `addPayment(debt.id, { amount, date, note?, tipAmount?, walletId? })` (`:1541`).
4. Determines transaction direction: **`they_owe` ⇒ income**, **`i_owe` ⇒ expense** (`:1554`).
5. Creates the linked Transaction:
   - Personal mode: `addTransaction({ amount, category, description, type, mode, walletId, inputMethod:'manual' })` (`:1563`), then **manually** adjusts wallet: income ⇒ `addToWallet`, expense ⇒ `deductFromWallet` (`:1573–1576`).
   - Business mode: `addBusinessTransaction({ ... type: income|cost ... })` (`:1578`), same wallet adjustment (`:1586–1589`).
6. Back-links both sides: `updatePayment(debt.id, paymentId, { linkedTransactionId })`, and on the transaction sets `{ linkedPaymentId, linkedDebtId }` (`:1592–1604`).
7. If the payment settles a split- or shared-sub-linked debt, marks the participant/member paid (`markSplitParticipantPaid` / `markSharedSubPayment`, `:1608–1613`).

The payment category defaults: `they_owe` ⇒ `debt_paid`, `i_owe` ⇒ `debt_payment` (`openPaymentModal`, `:1514`).

**Answer to the explicit questions:** Yes — a payment (a) optionally deducts/adds a wallet (only if a `walletId` is selected), and (b) creates a linked Transaction (personal `Transaction` or `BusinessTransaction`) carrying `linkedDebtId` + `linkedPaymentId`. The debt's `Payment.linkedTransactionId` points back. Cite: `DebtTracking.tsx:1561–1605`.

### 3.5 Tip handling
If `amount > remaining`, the UI raises a tip-confirm modal (`handleRecordPayment`, `:1834–1841`, `setTipConfirmData`). On confirm, the full amount flows but the debt only counts `remaining` toward `paidAmount` (store cap) with the excess recorded as `tipAmount` and appended to the transaction description `(incl. tip RM x.xx)` (`:1559`).

### 3.6 Consolidated / group payment (multi-debt, one cash event)
`handleRecordPayment` group branch, `DebtTracking.tsx:1652–1828`. When `groupPaymentId` is set (paying all of a person's debts at once):
- Gathers all unsettled, non-archived debts for the contact, oldest first (`:1655–1657`).
- **Netting**: if the group has both `i_owe` and `theyOwe` remaining (`isMixed`) AND the full net is being paid, it offsets the smaller side against the larger using internal "netted" payments (note `'netted'`), settling the offset (`:1678–1719`). If `iOweRem === theyOweRem`, everything nets to zero and settles with no cash (`:1712–1718`).
- A single linked Transaction is created for the cash leg (`:1733–1748`); its direction is the net direction.
- The cash is **distributed proportionally** across the net-direction debts so every debt shows progress (no early settlement), with rounding to the last debt (`:1759–1793`). All resulting payments share the same `linkedTransactionId` (marking them a consolidated batch).
- Leftover/overpayment is attached as a tip to the last payment (`:1796–1808`).
- The transaction's reverse link (`linkedDebtId`, `linkedPaymentId`) points to the **first** debt (`:1810–1814`).

### 3.7 Editing a payment
`handleSavePayDetail` → `commitPayDetailSave` (`DebtTracking.tsx:1934–2052`). Guards:
- Blocks amount edits on **consolidated** payments (siblings share `linkedTransactionId`) — `:1945–1955`.
- Blocks amount edits on a **settled** debt (would silently un-settle) — `:1958–1963`.
- If new amount overpays, raises tip-confirm (`:1965–1974`).
- On commit: `updatePayment(...)` (store appends a `PaymentEdit` if amount/note changed). Syncs the linked transaction amount (`updateTransaction`/`updateBusinessTransaction`). Wallet sync is **skipped in personal mode** (the transaction update already moved the wallet); in business mode it applies the delta directly (`:2019–2033`). If the edit un-settles a split/shared-sub debt, it unmarks the participant (`:2036–2047`).

Store `updatePayment` (`debtStore.ts:200–240`): recomputes paid/status; appends `PaymentEdit { editedAt, previousAmount, previousNote }` only when amount or note actually changed.

### 3.8 Deleting a payment
`handleDeletePayment` (`DebtTracking.tsx:1846–1919`):
- Detects consolidated siblings (same `linkedTransactionId`); if present, prompts to remove all (`:1851–1868`).
- Deletes the linked transaction **once** (`:1877–1880`).
- Reverses wallet **per payment**, but **only in non-personal mode** — in personal mode `deleteTransaction` already reversed the wallet (`:1882–1891`). Reversal direction: `they_owe` ⇒ `deductFromWallet`, `i_owe` ⇒ `addToWallet`.
- If a payment was the one that settled a split/shared-sub debt, unmarks the participant (`:1894–1910`).
- Calls store `deletePayment` for each.

Store `deletePayment` (`debtStore.ts:165–198`): removes payment, recomputes paid/status, and **itself** refunds the wallet for that payment's `walletId` via `addToWallet(walletId, amount)` (`:195–197`). NOTE: this store-level refund is in addition to the screen-level logic; the screen guards against double-reversal in personal mode by relying on `deleteTransaction`. (See §10 gotchas.)

## 4. EDIT AUDIT TRAIL (`editLog`)

Two append-only logs, matching the project "Edit Audit Trail Pattern":

### 4.1 Debt-level (`DebtEdit[]`)
Recorded in `updateDebt` (`debtStore.ts:61–102`). Appends an entry for each changed field among `totalAmount`, `description`, `contact`:
```
{ editedAt, field, previousValue, newValue }
```
Only appends if the value actually changed (`:67–75`). When `totalAmount` changes it also recomputes `paidAmount` (capped) and re-derives status (`:84–98`).

### 4.2 Payment-level (`PaymentEdit[]`)
Recorded in `updatePayment` (`debtStore.ts:200–222`). Snapshots **before** applying:
```
{ editedAt, previousAmount, previousNote }
```
Appended only when amount or note changed (`:210–214`).

### 4.3 UI
- **Bronze "edited" badge** on a payment row in history: `DebtTracking.tsx:6788–6792` — `edited {format(lastEdit.editedAt, 'MMM d, HH:mm')}`.
- **Edit History section** in the payment detail sheet: `DebtTracking.tsx:6179–6196` — header `t.debts.editHistory`, count `N changes`, rows newest-first (`[...editLog].reverse()`), each row showing `was RM {previousAmount}`.
- Help copy: `DebtTracking.tsx:8604` — "edit tracking — payment edits are logged. look for the 'edited' badge on modified payments."

### 4.4 Persistence
`partialize` serializes `editLog.editedAt` to ISO on both debts and payments (`debtStore.ts:551–559`); `onRehydrateStorage` revives them with the safe-date helper `sd()` (`debtStore.ts:612–623`).

### 4.5 Sync rule
The memory rule "initiator handles wallet adjustment; synced side just patches the amount" applies to the **cross-device personal sync** (§9) and the transaction↔payment bridge (§8). The party that initiates a payment/edit performs the wallet movement; the linked transaction merely mirrors the amount. There is **no two-user debt sharing** in this codebase, so there is no second human "synced side" — see §9.

## 5. STATUSES & LIFECYCLE

### 5.1 Derivation (single source of truth)
Status is always recomputed from `paidAmount` vs `totalAmount`, never set independently. Logic appears identically in `addPayment` (`debtStore.ts:148–151`), `updateDebt` (`:91–97`), `deletePayment` (`:179–184`), `updatePayment` (`:225–230`):
```
paidAmount >= totalAmount → 'settled'
paidAmount > 0           → 'partial'
else                     → 'pending'
```
`paidAmount` is always `min(totalAmount, sum(payments))` — overpayment never inflates it.

### 5.2 Lifecycle
`pending` (created, no payments) → `partial` (some paid) → `settled` (fully paid). Reaching `settled` triggers split/shared-sub participant marking (§3.4).

### 5.3 Settle / resolve
There is no separate "settle" toggle on the store; settlement is reached by recording a payment that covers the remainder. The payment modal can pre-fill the full remaining amount (e.g. split mark-paid pre-fills `remaining`, `:2070–2072`). Group netting can settle without cash (§3.6).

### 5.4 Archive (soft hide, non-destructive)
`archiveDebt`/`unarchiveDebt` (`debtStore.ts:112–124`) set `isArchived` + `archivedAt`. Archived debts are excluded from group/consolidated operations and reject new payments (`openPaymentModal`/`processPayment` guards, `:1505`, `:1525`).

### 5.5 Delete and what reverses
`handleDeleteDebt` (`DebtTracking.tsx:1460–1500`):
- **Blocks** deletion if the debt has consolidated-batch payments shared with siblings ("Remove the group payment first") — `:1464–1477`.
- **Blocks** deletion if `splitId` is set ("Edit or delete the split instead") — `:1479–1486`.
- Otherwise confirms, then `cleanupDebtPayments(debt)` + `deleteDebt(id)`.

`cleanupDebtPayments` (`:1434–1458`): for each payment, deletes the linked transaction (unless siblings exist), and reverses the wallet — **skipping personal mode when the transaction was deleted** (deleteTransaction already reversed). Reversal: `they_owe` ⇒ deduct, `i_owe` ⇒ add.

Store `deleteDebt` (`debtStore.ts:104–110`): removes the debt, records id in `_deletedDebtIds`, and registers a durable tombstone (`useTombstoneStore.addTombstones([id])`) to prevent re-sync resurrection.

`deleteContact` (`debtStore.ts:524–539`): does NOT delete that person's debts/splits — instead it renames the contact to `'(deleted)'` inside affected debts and split participants, preserving financial history; tombstones the contact.

## 6. SPLITS (splitting an expense among people)

### 6.1 Model — `SplitExpense`
`src/types/index.ts:901–922`. Fields: `id`, `description`, `totalAmount`, `splitMethod` (`'equal'|'custom'|'item_based'`, type `:829`), `participants: SplitParticipant[]`, `items: SplitItem[]`, `paidBy?: Contact`, `category?`, `taxAmount?`, `taxHandling?` (`'divide'|'waive'`, `:830`), `linkedTransactionId?`, `walletId?`, `mode`, `createdAt`, `updatedAt`, `status?: 'draft'|'final'`, `draftReceipt?`, `isArchived?`, `archivedAt?`.
- `SplitParticipant` (`:889`): `{ contact, amount, isPaid }`.
- `SplitItem` (`:895`): `{ name, amount, assignedTo: Contact[] }` (item-based assignment, shares divided by count).

### 6.2 How a split becomes debts (Split → Debt linking)
Save handlers: simple split editor `DebtTracking.tsx:2445–2530` (create) / `:2330–2444` (edit), and the receipt **wizard** `:3030–3135`.

Direction logic based on `paidBy` (self pseudo-contact `'__self__'`):
- **If I (`__self__`) paid** (`:2461`, `:3058`): create an **expense** transaction for the full `totalAmount` (personal `Transaction` or `BusinessTransaction`), deduct the wallet (or `useCredit` for credit wallets), back-link `linkedTransactionId` to the split, then for every other participant with `amount > 0`, `addDebt({ type: 'they_owe', totalAmount: p.amount, splitId, ... })` (`:2500–2512`, `:3102–3114`).
- **If someone else paid** (`:2513`, `:3115`): find my share (`__self__` participant) and create a single `addDebt({ type: 'i_owe', totalAmount: myShare, splitId, ... })` to that payer — **no expense transaction / no wallet change for me** (`:2516–2526`, `:3118–3128`).

Each generated debt carries `splitId` linking it back to the `SplitExpense`. The split's `participants[].isPaid` is the human-facing status; the linked debt's `status` is the financial truth.

### 6.3 Marking a split participant paid
`handleSplitMarkPaid` (`DebtTracking.tsx:2055–2084`):
- If a linked debt exists and is unsettled, it **routes through the real payment infrastructure** — opens the payment modal pre-filled with the remaining amount and a `Split: {desc}` note (`:2065–2076`). This creates a real Transaction + wallet movement.
- If no linked debt (a purely manual split with no debt), it's a simple boolean toggle `markSplitParticipantPaid` (`:2079`).

`handleSplitUndoPaid` (`:2086–`): reverses ALL recorded payments for that participant's linked debt (deleting transactions, reversing wallets with the personal-mode skip rule), then `unmarkSplitParticipantPaid` (`:2144`).

Store: `markSplitParticipantPaid`/`unmarkSplitParticipantPaid` (`debtStore.ts:289–315`) just flip `participants[].isPaid`.

### 6.4 Editing a split
`:2330–2442`. Handles: removed participants (deletes their linked debts + reverses), `paidBy` change (flips direction — tears down and recreates transaction/debts, `:2369–2394`), total change (cascades to transaction amount + wallet delta + per-participant debt `totalAmount`), with a guard that **won't shrink** a participant's debt below already-paid amount or change a settled one (`settledAmountBlocked`, `:2427–2441`).

### 6.5 Deleting a split
`handleDeleteSplit` (`:2563–2593`): drafts delete cleanly (no side effects). Final splits run `cleanupSplitTransaction` (deletes the expense transaction, reverses wallet/credit, `:2537–2561`) and `cleanupDebtPayments` + `deleteDebt` for every linked debt.

### 6.6 Receipt-scan wizard & WhatsApp request
- The item-based wizard builds `splitMethod: 'item_based'` with optional `taxAmount`/`taxHandling` and can be saved as a draft (`status: 'draft'`) before finalizing (`:3137`+).
- `composePaymentMessage` (`:3316–3352`) generates a WhatsApp dunning message with a per-item breakdown drawn from the linked split's `items` and tax-per-person, optionally attaching a payment QR (`sendWhatsAppWithQr`, `:3387`).

### 6.7 Store split actions
`addSplit`/`updateSplit`/`deleteSplit`/`archiveSplit`/`unarchiveSplit` (`debtStore.ts:242–287`). `deleteSplit` tombstones the id.

## 7. SHARED SUBSCRIPTION (Netflix/Spotify cost-sharing)

**Status: fully BUILT** (not just a blueprint). It is the third tab in the debt area, with form, detail, and price-change sheets. (The memory note "Shared Subscription Tracker — blueprint approved, build later" is stale; the feature is implemented.)

### 7.1 Models — `src/types/index.ts:926–973`
- `SharedSubscription`: `{ id, name, iconName?, imageUri?, totalAmount, billingCycle: 'monthly'|'quarterly'|'yearly', billingDay, members: SharedSubMember[], monthRecords: SharedSubMonthRecord[], priceHistory: SharedSubPriceChange[], subscriptionId?, isActive, note?, createdAt, updatedAt }`.
- `SharedSubMember`: `{ contact, tag, shareAmount, isActive, joinedAt }` (tag e.g. "owner", "profile 2").
- `SharedSubMonthRecord`: `{ month: 'YYYY-MM', totalAmount, payments: SharedSubPayment[], debtsGenerated: boolean }`.

### 7.2 Store actions — `debtStore.ts:317–511`
- `addSharedSubscription` (`:317`): seeds `monthRecords: []` and an initial `priceHistory` entry from the current `totalAmount`/member shares.
- `ensureMonthRecord` (`:391`): lazily creates a month record using the **effective price** (latest `priceHistory` entry with `effectiveFrom <= month`), populating per-active-member share amounts; sets `debtsGenerated: false`.
- `markSharedSubPayment`/`unmarkSharedSubPayment` (`:423–461`): flip per-member `isPaid` + `paidAt` within a month record.
- `recordSharedSubPriceChange` (`:463`): updates `totalAmount` + member `shareAmount`, appends a `priceHistory` entry (forward-effective).
- `updateMonthAmounts` (`:480`): retroactively edits one month's total + member shares AND cascades to the linked debts for that `sharedSubId`+`sharedSubMonth` (`:505–510`).
- `deleteSharedSubscription` (`:348`): removes the sub AND all debts with that `sharedSubId` (`:352`), tombstones the id.
- Member ops: `addSharedSubMember`, `updateSharedSubMember`, `removeSharedSubMember` (soft — sets `isActive: false`).

### 7.3 Debt generation from a month
`handleGenerateSharedDebts(subId, month)` (`DebtTracking.tsx:3428–3464`): ensures the month record, then for each member payment (skipping `'__self__'`) with no existing debt, `addDebt({ type: 'they_owe', totalAmount: payment.amount, description: '{subName} - {MMM yyyy}', sharedSubId, sharedSubMonth, mode })`. Marks `record.debtsGenerated = true` via `useDebtStore.setState`. Settling such a debt marks the member paid (§3.4); deleting the settling payment unmarks (§3.8).

## 8. LINKED TRANSACTIONS (`linkedDebtId`)

### 8.1 The bridge
A debt **payment** creates a `Transaction` (personal) or `BusinessTransaction` (business). `Transaction.linkedDebtId` (`src/types/index.ts:592`) + `linkedPaymentId` (`:591`) point to the debt and payment; the `Payment.linkedTransactionId` points back. So debt payments appear in the normal transaction list as ordinary income/expense rows (income for `they_owe`, expense for `i_owe`). Wired at `DebtTracking.tsx:1592–1605`.

`BusinessTransaction` carries the same `linkedDebtId`/`linkedPaymentId` (`src/types/index.ts:33–34`).

### 8.2 Protection rules (transaction side)
Debt-linked transactions are protected so the two sides cannot desync:
- **Amount locked** in the edit sheet: `EditTransactionSheet.tsx:357` `isLinkedDebt = !!transaction.linkedDebtId`; `:362` `isAmountLocked = isLinkedDebt || isLinkedGoal || isTransferLinked`. When debt-linked, only description + tags are editable (`:566–574`).
- **If an amount edit is somehow allowed**, the change is **pushed back into the debt** rather than the wallet: `EditTransactionSheet.tsx:634–645` calls `useDebtStore.getState().updatePayment(linkedDebtId, linkedPaymentId, { amount: newAmount })` (or finds the payment by `linkedTransactionId`).
- **Deletion blocked**: swipe-delete and bulk-delete in `TransactionsList.tsx:437`, `:506`, and single delete in `EditTransactionSheet.tsx:669` all bail with toast `t.transactionList.debtLinkedCannotDelete` ("manage from the debt screen"). To remove the money movement you must delete the payment from the debt screen, which cascades to the transaction.

### 8.3 Other consumers
`linkedDebtId` referenced in `Dashboard.tsx`, `TransactionItem.tsx` (badge/label), and `exportService.ts` (CSV/export tagging).

## 9. CROSS-PERSON / CROSS-DEVICE SYNC

**There is NO two-user shared-debt feature.** A debt lives entirely on one user's device(s). The other party is just a `Contact` (name/phone) — they have no account link, and nothing is pushed to them.

What DOES sync is **single-user, multi-device backup/restore** via `src/services/personalSync.ts` against Supabase (the user's own account):
- `debtToRemote` (`:169`) maps a `Debt` to row `{ user_id, local_id, contact_name, contact_phone, type, total_amount, paid_amount, status, payments (JSON, dates ISO-ified, editLog preserved), due_date, note, wallet_local_id, updated_at }`. Tables: `personal_debts`, `personal_splits`, `personal_contacts` (`pullTable` calls at `:549–551`).
- `splitToRemote` (`:192`) maps splits.
- Pull merges by id with **last-write-wins** on `updatedAt` and respects durable tombstones (`mergeById`, `:572–584`; applied at `:598–602`).
- **Notably absent from sync**: `sharedSubscriptions` are not in `personalSync` — shared subscriptions are **local-only** (no cloud backup).

**Who owns the wallet adjustment:** the device/user that records the payment performs the wallet movement and creates the linked transaction; the cloud row is a passive mirror keyed by `local_id`. There is no peer to reconcile against. (Contrast: the seller-mode Supabase backend in MEMORY.md is unrelated — `sellerSync.ts` has no debt references.)

## 10. EDGE CASES, GOTCHAS, WHAT'S MANUAL

1. **`pending` vs "unpaid"**: the status literal is `pending`, displayed "Pending"; there is no `unpaid` literal. The unpaid state == `pending`.
2. **Overpayment is capped in the store**: `addPayment` caps to remaining; the excess is only a UI `tipAmount` + appended to the transaction description. The debt never reads as overpaid (`debtStore.ts:135`, `148`).
3. **Double wallet-reversal risk on payment delete**: store `deletePayment` itself refunds the wallet (`debtStore.ts:195–197`), AND the screen's delete/undo flows reverse wallets too. The screen compensates by skipping its own reversal in personal mode when `deleteTransaction` already reversed (and `deleteTransaction` reverses the wallet — `personalStore.ts:105–108`). This is a fragile coordination point; the store-level refund is keyed on `payment.walletId` which group/netted payments may not carry.
4. **Group netting only on full net**: mixed-direction netting happens only when the full net amount is paid; partial mixed payments pay only the net-direction debts (`DebtTracking.tsx:1677–1726`).
5. **Consolidated payments are semi-immutable**: amount edits are blocked on any payment whose `linkedTransactionId` has siblings; you must remove and re-record (`:1945–1954`). A debt with consolidated siblings also can't be deleted directly (`:1464–1477`).
6. **Split-linked debts can't be deleted individually** — must edit/delete the split (`:1479–1486`).
7. **Settled-debt amount edits blocked** to avoid silent un-settling (`:1958–1963`); split edit similarly won't shrink a paid/settled participant debt (`settledAmountBlocked`, `:2427–2441`).
8. **`__self__` is a magic contact id** representing "me" in splits and shared-subs; debts are never created for `__self__` (`:2386`, `:2501`, `:3438`).
9. **Credit wallets**: when I pay a split bill from a credit wallet, the code calls `useCredit`/`repayCredit` instead of deduct/add (`:2493`, `:2556`, `:3094`).
10. **Shared subscriptions are local-only** (not in `personalSync`); losing the device loses them.
11. **`groupId` auto-grouping** keys on `contact.id || contact.name`; a contact renamed to `(deleted)` after `deleteContact` can collide other deleted contacts into one group. Legacy debts without `groupId` are migrated on rehydrate (`debtStore.ts:596–604`).
12. **Manual steps (no automation)**: recording every payment, choosing the wallet, generating shared-sub debts per month (`handleGenerateSharedDebts` is user-triggered), marking split participants, sending WhatsApp reminders, and recording price changes are all explicit user actions. There is no auto-detection of repayments from bank/transaction data.
13. **Date safety**: all stored dates are ISO-serialized in `partialize` and revived with a guarded `sd()` helper in `onRehydrateStorage` to avoid `Invalid time value` crashes (`debtStore.ts:544–657`).
